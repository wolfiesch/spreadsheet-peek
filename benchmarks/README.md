# Benchmarks

Token-cost measurements for the claims in the main README.

## Methodology

We measure the token cost of `wolfxl peek`'s output modes using `tiktoken`'s `cl100k_base` encoding (the GPT-4 tokenizer). This is a proxy for Claude's tokenizer - the absolute numbers differ slightly, but the **ratios between modes are stable** across tokenizers because they're driven by the same structural properties (how many Unicode box-drawing chars, how much repeated whitespace).

Measurements cover three sample workbooks across the shape spectrum:

- [`examples/sample-financials.xlsx`](../examples/sample-financials.xlsx) - 7 columns, typical financial-report shape. Generated by [`examples/generate_sample.py`](../examples/generate_sample.py).
- [`examples/tall-ledger.xlsx`](../examples/tall-ledger.xlsx) - 8 columns, 120 data rows, GL-detail shape with dates, memo text, and running balances. Generated by [`examples/generate_tall_ledger.py`](../examples/generate_tall_ledger.py).
- [`examples/wide-table.xlsx`](../examples/wide-table.xlsx) - 29 columns, an operations-dashboard stress test. Generated by [`examples/generate_wide_table.py`](../examples/generate_wide_table.py).

Delimited-input costs are measured separately so the workbook output-mode ratios stay stable:

- [`examples/sample-ledger.csv`](../examples/sample-ledger.csv) - 7-column ledger extract. Generated by [`examples/generate_delimited_samples.py`](../examples/generate_delimited_samples.py).
- [`examples/sample-ledger.tsv`](../examples/sample-ledger.tsv) - the same rows as TSV, used to confirm direct `.tsv` cost parity.

Reproduce with:

```bash
uv run --with tiktoken --with openpyxl python benchmarks/measure_tokens.py
```

**Key ratios**:
- Financials: box-drawing costs ~3.9x more tokens per row than text export (5-row), ~3.7x at 15 rows.
- Tall ledger: box-drawing costs ~3.6x per row (5-row), ~3.0x at 15 rows. This is the accounting-detail middle case between financial statements and very wide operational tables.
- Wide: box-drawing costs ~3.0x per row (5-row), ~2.8x at 15 rows. The *ratio* is smaller because text-export baseline is larger per row; the *absolute* per-row savings (299 tokens vs 85-90 for narrower samples) is ~3.5x higher.

CSV and TSV are byte-for-byte equivalent after `wolfxl` parses them into the same synthetic sheet, so their direct-input token costs match on this fixture. These rows are deliberately separate from the workbook table above: the workbook table compares output modes across `.xlsx` shapes, while this table proves the direct delimited path has its own bounded cost profile.

## Why these numbers matter

An agent that previews 10 spreadsheets during a session at 15 rows each, assuming a mix:

- **Box-drawing everywhere (financials-shaped)**: 10 × 1,313 = **13,130 tokens** just on previews.
- **Box-drawing everywhere (ledger-shaped)**: 10 × 1,434 = **14,340 tokens**.
- **Box-drawing everywhere (wide-shaped)**: 10 × 5,623 = **56,230 tokens** - roughly 28% of a 200K window on previews alone.
- **Mode switch (box first, text after)**: 1,313 + (9 × 357) = **4,526 tokens** on the financials path; 1,434 + (9 × 479) = **5,745 tokens** on the ledger path; 5,623 + (9 × 1,975) = **23,398 tokens** on the wide path.

The wide-table case is where this matters most in absolute terms: a single naive 15-row preview already costs more than four financial-shape previews. On a workbook wider than ~25 columns, skip box-drawing entirely and go straight to `--export text | sed -n '1,Np'`.

## What's not measured (yet)

- **Claude tokenizer ground truth**: Anthropic's tokenizer isn't publicly available, so we use cl100k_base as a proxy. PRs welcome if Anthropic publishes a client-side tokenizer.
- **Generation-side token cost**: The box-drawing output is what the *agent reads* (context tokens). The *generation cost* of writing `wolfxl peek file -n 15` vs writing a 20-line Python script is a separate measurement - we estimate ~150-200 tokens saved per invocation on the generation side.
- **Comma-delimited `.txt` input costs**: `wolfxl peek` 0.8.0 treats `.txt` as comma-delimited input. The CSV table above is the closest proxy; add a `.txt` fixture only if we need to document a distinct text-table shape.

## Claim Verification

Behavioral claims are checked separately:

```bash
uv run --with openpyxl python benchmarks/verify_claims.py
```

That script smoke-tests direct previews for `.xlsx`, `.xlsm`, `.xls`, `.xlsb`, `.ods`, `.csv`, `.tsv`, and comma-delimited `.txt`; Balance Sheet date rendering; currency and percentage number-format rendering; `sed`-limited text export; and `wolfxl agent --max-tokens`.

## Contributing benchmark data

If you have an interesting workbook shape (very wide, very tall, CSV instead of xlsx), open a PR adding it to `examples/` and rerun `measure_tokens.py` - the results table here should grow over time.
