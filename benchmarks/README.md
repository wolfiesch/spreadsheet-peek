# Benchmarks

Token-cost measurements for the claims in the main README.

## Methodology

We measure the token cost of `wolfxl peek`'s output modes using `tiktoken`'s `cl100k_base` encoding (the GPT-4 tokenizer). This is a proxy for Claude's tokenizer - the absolute numbers differ slightly, but the **ratios between modes are stable** across tokenizers because they're driven by the same structural properties (how many Unicode box-drawing chars, how much repeated whitespace).

Measurements cover four sample workbooks across the shape spectrum:

- [`examples/sample-financials.xlsx`](../examples/sample-financials.xlsx) - 7 columns, typical financial-report shape. Generated by [`examples/generate_sample.py`](../examples/generate_sample.py).
- [`examples/tall-ledger.xlsx`](../examples/tall-ledger.xlsx) - 8 columns, 120 data rows, GL-detail shape with dates, memo text, and running balances. Generated by [`examples/generate_tall_ledger.py`](../examples/generate_tall_ledger.py).
- [`examples/wide-table.xlsx`](../examples/wide-table.xlsx) - 29 columns, an operations-dashboard stress test. Generated by [`examples/generate_wide_table.py`](../examples/generate_wide_table.py).
- [`examples/messy-ops-export.xlsx`](../examples/messy-ops-export.xlsx) - 12 columns, 244 physical rows, title and note rows before the real headers, long wrapped memos, dates, currency, percentages, punctuation, blanks, and multiple sheets. Generated by [`examples/generate_messy_workbook.py`](../examples/generate_messy_workbook.py).

Delimited-input costs are measured separately so the workbook output-mode ratios stay stable:

- [`examples/sample-ledger.csv`](../examples/sample-ledger.csv) - 7-column ledger extract. Generated by [`examples/generate_delimited_samples.py`](../examples/generate_delimited_samples.py).
- [`examples/sample-ledger.tsv`](../examples/sample-ledger.tsv) - the same rows as TSV, used to confirm direct `.tsv` cost parity.
- [`examples/sample-ledger.txt`](../examples/sample-ledger.txt) - the same comma-delimited rows with a `.txt` extension, used to prove the documented text-table path.
- [`examples/quoted-multiline.csv`](../examples/quoted-multiline.csv) - a compact CSV with quoted commas, escaped quotes, tabs, and embedded newlines.

Reproduce with:

```bash
uv run --with tiktoken --with openpyxl python benchmarks/measure_tokens.py
```

## Local comparison harness

Use this when you want to compare Spreadsheet Peek output size against other
local converters on your machine:

```bash
uv run --with tiktoken --with openpyxl python benchmarks/compare_converters.py
uv run --with tiktoken --with openpyxl --with 'markitdown[xlsx]' --with agent-xlsx --with click python benchmarks/compare_converters.py --file examples/wide-table.xlsx
```

The required Spreadsheet Peek rows are text export, workbook map, and bounded
agent summary. The harness also measures a built-in vanilla `openpyxl` tuple
dump when `openpyxl` is installed. Optional modes currently run only when their
commands are installed: MarkItDown (`markitdown`) and `agent-xlsx`. Missing
optional tools are reported as skipped. Treat the output as a local comparison
harness, not a universal performance claim.

The fair claim is narrow: Spreadsheet Peek is a small first-look layer. It is
excellent when you need a cheap preview habit, a bounded summary, or a workbook
map. MarkItDown is broader whole-file conversion. `agent-xlsx probe` is a strong
structure-first comparator. A tiny openpyxl tuple dump is sometimes compact in
output tokens, but the agent still has to generate code, wait for Python import
startup, and read raw tuples instead of formatted spreadsheet values.

## Messy workbook growth curve

The messy workbook answers a slightly different question: does the advantage get
stronger as the preview grows? The answer is **yes in absolute tokens, no as an
ever-increasing multiplier**. As more real data rows enter the preview, text
export carries more cell content too, so the ratio settles down. The total saved
context still grows quickly.

| Mode | Tokens | Bytes | Rows | Tokens/row |
|------|-------:|------:|-----:|-----------:|
| Messy ops export (12 cols) - Box preview (5 rows) | 798 | 7,274 | 5 | 159.6 |
| Messy ops export (12 cols) - Text export (5 rows) | 154 | 606 | 5 | 30.8 |
| Messy ops export (12 cols) - Box preview (15 rows) | 2,220 | 18,634 | 15 | 148.0 |
| Messy ops export (12 cols) - Text export (15 rows) | 745 | 2,351 | 15 | 49.7 |
| Messy ops export (12 cols) - Box preview (50 rows) | 7,109 | 56,784 | 50 | 142.2 |
| Messy ops export (12 cols) - Text export (50 rows) | 2,798 | 8,425 | 50 | 56.0 |

**Growth ratios**:
- 5 rows: box preview is 5.2x more tokens than text export, saving 644 tokens.
- 15 rows: box preview is 3.0x more tokens than text export, saving 1,475 tokens.
- 50 rows: box preview is 2.5x more tokens than text export, saving 4,311 tokens.

This makes the public claim stronger without overstating it: Spreadsheet Peek's
mode switch saves more total context as previews grow, while the exact
multiplier depends on workbook shape and how much real cell content is present.

CSV, TSV, and comma-delimited TXT are byte-for-byte equivalent after `wolfxl` parses them into the same synthetic sheet, so their direct-input token costs match on this fixture. The quoted multiline CSV proves the CSV parser path against a shape that naive `head`/`column -s, -t` previews mishandle. These rows are deliberately separate from the workbook table above: the workbook table compares output modes across `.xlsx` shapes, while this table proves the direct delimited path has its own bounded cost profile.

## Why these numbers matter

An agent that previews 10 spreadsheets during a session at 15 rows each, assuming a mix:

- **Box-drawing everywhere (financials-shaped)**: 10 × 1,313 = **13,130 tokens** just on previews.
- **Box-drawing everywhere (ledger-shaped)**: 10 × 1,434 = **14,340 tokens**.
- **Box-drawing everywhere (wide-shaped)**: 10 × 5,623 = **56,230 tokens** - roughly 28% of a 200K window on previews alone.
- **Mode switch (box first, text after)**: 1,313 + (9 × 357) = **4,526 tokens** on the financials path; 1,434 + (9 × 479) = **5,745 tokens** on the ledger path; 5,623 + (9 × 1,975) = **23,398 tokens** on the wide path.

The wide-table case is where this matters most in absolute terms: a single readable 15-row box preview already costs more than four financial-shape previews. On a workbook wider than ~25 columns, skip box-drawing entirely and go straight to `--export text | sed -n '1,Np'`.

Markdown export sits between text export and box-drawing in token cost. Use it when another tool or context converter benefits from Markdown tables; keep text export as the repeat-preview default.

## What's not measured (yet)

- **Claude tokenizer ground truth**: Anthropic's tokenizer isn't publicly available, so we use cl100k_base as a proxy. PRs welcome if Anthropic publishes a client-side tokenizer.
- **Generation-side token cost**: The box-drawing output is what the *agent reads* (context tokens). The *generation cost* of writing `wolfxl peek file -n 15` vs writing a 20-line Python script is a separate measurement - we estimate ~150-200 tokens saved per invocation on the generation side.
- **Custom-delimiter input costs**: `wolfxl peek` 0.9.0 treats `.txt` as comma-delimited input and `.tsv` as tab-delimited input. Arbitrary delimiter detection is outside the direct path; use the fallback tooling in `SKILL.md` for pipe-delimited or semicolon-delimited files.

## Claim Verification

Behavioral claims are checked separately:

```bash
uv run --with openpyxl python benchmarks/verify_claims.py
```

That script smoke-tests direct previews for `.xlsx`, `.xlsm`, `.xls`, `.xlsb`, `.ods`, `.csv`, `.tsv`, and comma-delimited `.txt`; Balance Sheet date rendering; currency and percentage number-format rendering; `sed`-limited text export; and `wolfxl agent --max-tokens`.
